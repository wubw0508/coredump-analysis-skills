# Analysis Runbook

Use this when preparing, launching, monitoring, or recovering coredump analysis runs.

## Prerequisites

Use the loaded skill root as `$SKILLS_DIR` and avoid machine-specific paths:

```bash
cd "$SKILLS_DIR"
python3 check_skill_sync.py  # after repo-managed docs/reference edits
```

The repo checkout is the distributable source of truth. Do not sync to a user-private Hermes cache.

`accounts.json` must contain real values for `metabase.account`, `gerrit.account`, `shuttle.account`, and `system.sudo_password`:

```bash
grep -c '在此处' "$SKILLS_DIR/accounts.json"  # 0 = configured
ls -la ~/.ssh/id_rsa                         # when clone/push is involved
```

## Package scope

`packages.txt` is the default full-analysis scope. Entries may be direct package names, `project:package`, or `project:package branch` mappings. Metabase download must always use the package side; project context is only for source/Gerrit/fix mapping.

## Entrypoints

```bash
# Full analysis over packages.txt
bash run_analysis_agent.sh --background --progress 180

# Single package
bash run_analysis_agent.sh --packages dde-dock --background

# Single package with date range
bash run_analysis_agent.sh --packages dde-dock --start-date 2026-03-14 --end-date 2026-04-14

# Lower-level single-package flow
bash coredump-full-analysis/scripts/analyze_crash_complete.sh --package dde-dock
```

Important defaults:

- auto-fix-submit is enabled by default unless disabled by environment
- `analyze_crash_complete.sh --max-crashes` defaults to `0` (all deduplicated crashes)
- enhanced addr2line defaults to `300` frames; automatic deep dive uses at least `600`

## Workspace first stops

Typical workspace contains `1.数据下载/`, `2.数据筛选/`, `3.代码管理/`, `4.包管理/`, `5.崩溃分析/`, `6.修复补丁/`, and `6.总结报告/`.

Read workspace rollups before package logs:

- `6.总结报告/analysis_summary_final.md` — human-facing final summary generated by `generate_workspace_summary.py` after each analysis/summary refresh
- `6.总结报告/package_status.tsv`
- `6.总结报告/version_status.tsv`
- `6.总结报告/run_manifest.json|md`
- `6.总结报告/retry_summary.md`
- `6.总结报告/auto_fix_overview.json|md`
- `6.总结报告/new_crashes_overview.json|md`

Monitoring:

```bash
cat <workspace>/6.总结报告/package_status.tsv
less <workspace>/6.总结报告/logs/analysis_<pkg>.log
grep -c 'completed' <workspace>/6.总结报告/package_status.tsv
grep -c 'running' <workspace>/6.总结报告/package_status.tsv
```

`run_analysis_agent.sh --background` may emit harmless non-child-process wait warnings; verify actual state from status TSV and logs.

## Summary refresh recovery

If status files show completed but summaries still look stale/running/zero-metric, regenerate the workspace summary:

```bash
python3 coredump-full-analysis/scripts/reporting/generate_workspace_summary.py \
  --workspace <workspace> \
  --status-file <workspace>/6.总结报告/package_status.tsv \
  --version-status-file <workspace>/6.总结报告/version_status.tsv \
  --date-range-label '全部可下载数据（不按日期过滤）'
```

Then re-check status TSVs plus `retry_summary.md`, `auto_fix_overview.md`, and `new_crashes_overview.md`.

## Operational pitfalls

- Empty CSV can be legitimate; inspect produced data before assuming package/project mismatch.
- Shell wrappers need `PYTHONPATH` to include sibling script directories.
- Define `SCRIPT_DIR` before deriving `SKILLS_DIR` in shell scripts.
- Let Gerrit `commit-msg` generate `Change-Id`; never inject fake all-zero IDs.
- Normalize duplicated `origin/origin/...`; record missing target branches structurally.
- Gerrit `no new changes` may mean an existing review; verify by Change-Id.
- Markdown-only `auto-fix/*` branches are not real source-fix candidates.
